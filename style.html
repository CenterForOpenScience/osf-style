Style Guide

Layout

Container

OSF uses a fixed container layout as opposed to fluid. This makes sure that the width does not continue to adjust past 1200px width. Your documents should be wrapper inside a div with class "container." 


    <div class="container">
    ...
    </div>
Bootstrap documentation

Grid

Using the grid properly is critical for the correct view of the OSF components on all devices. Within a container you should ALWAYS use grids for layout and horizontal alignment. Do not use margins, padding, float or tables for layout arrangements.


<div class="row">
    <div class="col-md-12">...</div>
</div>
<div class="row">
    <div class="col-md-4">...</div>
    <div class="col-md-3">...</div>
    <div class="col-md-5">...</div>
</div>
<div class="row">
    <div class="col-md-6">...</div>
    <div class="col-md-6">...</div>
</div>

How to use the grid:

  • Grid columns need to add up to 12.
  • Each total of 12 should be inside its own "row" div.
  • Plan out the layout of your component on paper before you start coding and then write the html code of the row structure first. This will alleviate the problem of columns not matching up or some div tags not closing.
  • Bootstrap grid is designed to fit columns together. If there is an unnecessary amount of space in between components, first check that your grid is properly built.
  • Bootstrap grids do not align heights of columns. If you need column heights to be equal you will need to manually set them or use other libraries.
  • Do not use grids to center align a component horizontally unless there is no other semantically meaningful way to do so.
Bootstrap documentation

Layout helper CSS classes

For margin and padding OSF has helper classes that provide a quick method to add granular layout without having to define these distances for every item.

.type + direction + size | Example: .m-t-xs

  • Type : "m" : margin or "p" : padding
  • Direction : "t" : top, "b" : bottom, "l" : left, "r" : right; "v" : vertical (top and bottom), "h" : horizontal (left and right) or omit this and apply to all sides; i.e. .m-xs;
  • Size : "xs": 5px, "sm": 10px, "md": 15px, "lg" : 25px, "xl" : 50px
Examples:

How to space elements:

  • Use margin helpers at top and bottom of divs to separate content.
  • Do NOT use HR until you tried all margin sizes for spacing. Is sections need to be further separated you can try HR tag.

Word break

One of the most common testing scenarios is to use single long words in layout elements. Most often these words will break the layout and can have usability consequences such as hiding buttons or other content. Most text is automatically applied word break rule sets including "p, em, b, a, td" tags. You can also use an separate class ".break-word"

Inline edits

You can provide users the opportunity to edit content "in place" meaning that the content switches to input elements to make changes and returns back to being the regular content when either changes are saved or dismissed. Use the "osf-editable" class for items that are editable inline.

Name: OSF Project Name 


    <p>Name :  <span class="osf-editable"> OSF Project </span></p>

When to use inline editing

  • USE if it's an isolated edit not part of a form.
  • DO NOT use within a form, it adds no value.
  • USE if the text being edited is part of a regular size text such as paragraph.
  • USE if the text being edited is not more than a few words (i.e. Don't edit long text this way)
  • USE if the edit is not critical (can be quickly changed again).

If you use inline edits make sure you use them in a similar way across the site!

Understanding Bootstrap colors

Success

Use cases:

  • Save buttons
  • Messages which indicate a successful action, such as server connection, or successful save
Primary

Use cases:

  • Any neutral BUT primary action item, that is neither good nor bad. This depends on the order of items.
  • When it is the only button on the page (the color coordinates better with overall theme)
Info

Use cases:

  • Help and info buttons and texts
  • Settings, options and details
Warning

Use cases:

  • Alerts that bring attention to a potentially problematic issue that is not yet causing an error but might
  • Buttons that don't have a full delete effect but should be alerting, for instance "Delete Some"
Danger

Use cases:

  • Error alerts
  • All Delete and remove buttons

Which icons to use?

Action Icon to use FA class
Toggling show/hide of content with up and down fa-angle-right, fa-angle-down
Deleting tabular or repeating data fa-times
Deleting stand alone items or important items fa-trash-o
Inline help buttons fa-question-circle

There is no standardization beyond these icons. Use of icons with or without buttons is very contexual to what the action is. Developers should err on the side of not using icons if in doubt and should not use icons outside their conventions (i.e. Link icons, save icons etc.)

Should we use icons with buttons?

YES if:
  • There are a lot of buttons, i.e in a toolbar
  • Icons are repeated in other sections of the page to show that they are related to button text
  • If narrow screens show only icons and not the text
  • The button is very widely used together with text in general user experience. I.e. Create, Delete, Edit type of single words get icons; but not "Create Registration"
NO if:
  • There is a stand alone button
  • Button text is not general enough to be natural (e.g. Make public)
  • The icon is not intuitive for that task, e.g.

If icons are use in buttons they should always be on the LEFT side

Guidelines for button text

  • Do not use OK, even if it makes sense with the question being asked. People skim to the button often and OK doesn't help understand the meaning quickly.
  • For modals that only provide information to the user, use a single button with text "Got it" that closes the modal. Also use the X on top right. Do not use a question mark as modal header.
  • Buttons should have verbs and should start with verbs, e.g. "Make public" instead of "Public" alone or "Visibility Setting".
  • The button does not need to repeat the header, e.g. "Add" is enough for the submit button for a modal that says "Add Contributors".
  • Be deliberate in your use of Cancel vs Close. Cancel is supposed to undo any potential visual or data change or to denote change of intention by the user (e.g. Don't Save). Close simply removes an element from view. Modals need a Close button and forms need a Cancel button. If the form has brought the user to a different page Cancel should bring them back to where they were.
  • Back should be used in an OSF guided activity, where the user has to do a series of steps, and wants to return to a previous step. For example, signing up for the Preregister Prize. If the path is user driven, for example, creating a project, use the Cancel button.
  • Don't abandon Cancel or Close buttons next to Save in favor of Dismiss buttons (x at the top right of content). Cancel is easier to find and click and the X button is more appropriate for deleting portions of content (e.g. deleting a single employment institution in the profile).

Form buttons

With forms don't use Submit. Use the word that the activity requires.

Action Text to use
Creating a new item Create
Editing the information of an existing item Save
Deleting an item Delete
Applying a setting (for instance add ons) Apply

Guidelines for Modal Text

  • Modal titles should generally be clear, concise sentence fragments.
  • Modal titles should be in sentence case (first letter and proper nouns capitalized).
  • Modal titles should include question marks (if they are questions), but no other punctuation.
  • Follow 'Guidelines for Button Text' as shown above.
  • A user should be able to skip the body of the modal and still understand their general modal options.
  • Body text should be in full sentences.
  • These guidelines are modeled after how Google handles “Alerts with title bars.” See their guide for examples.

Inputs

Use Bootstrap form styles in all forms as it is.

Text input

  • Add "noresize" class to remove the ability of the user to resize this box. 

<input type="text" class="form-control" placeholder="Text input">

Text area

  • Text area input should be used for long form inputs.
  • Add "noresize" class to remove the ability of the user to resize this box. 

<textarea class="form-control noresize" rows="3"></textarea>

Checkboxes and options

  • Use checkboxes for elements that can have multiple selections.
  • Use options for elements with single possible selection.
  • For short options text you should use ".checkbox-inline" or ".radio-inline" to keep everything in one line 

<div class="checkbox">
    <label>
        <input type="checkbox" value="">
        Option one is this and that—be sure to include why it's great
    </label>
</div>

<div class="checkbox disabled">
    <label>
        <input type="checkbox" value="" disabled>
        Option two is disabled
    </label>
</div>

<div class="radio">
    <label>
        <input type="radio" name="optionsRadios" id="optionsRadios1" value="option1" checked>
        Option one is this and that—be sure to include why it's great
    </label>
</div>

<div class="radio">
    <label>
        <input type="radio" name="optionsRadios" id="optionsRadios2" value="option2">
        Option two can be something else and selecting it will deselect option one
    </label>
</div>
<div class="radio disabled">
    <label>
        <input type="radio" name="optionsRadios" id="optionsRadios3" value="option3" disabled>
        Option three is disabled
    </label>
</div>

Selects

  • In general DO NOT USE Selects. Try to find an alternative solution
  • If you have fewer than 6 items consider using options input instead
  • Select works best with large amount of selection options
  • Alphabetize the option list if appropriate -- see more below

<select class="form-control">
<option>1</option>
<option>2</option>
<option>3</option>
<option>4</option>
<option>5</option>
</select>

Input and form tips:

  • Input elements will generally fill in the container unless size is specified. Try to combine row and column wrappers to maintain responsiveness or use inline and horizontal form styles shown in the Bootstrap documentation.
  • Avoid making form elements too wide, consider other options for layout if the window is too wide.
